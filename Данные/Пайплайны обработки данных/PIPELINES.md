# Алгоритмы пайплайнов обработки данных

Этот файл описывает все пайплайны из папки `Данные/Пайплайны обработки данных`.

## Общая логика инкрементальной обработки

Все пайплайны работают одинаково:

1. Находят все исходные Excel-файлы в своей исходной папке.
2. Игнорируют временные файлы Excel, начинающиеся с `~$`.
3. Считают SHA-256 хэш каждого исходного файла.
4. Если итоговый CSV уже существует, читают из него `source_file_hash` и `source_file_path`.
5. Если файл с таким хэшем уже есть в итоговой таблице, он пропускается.
6. Если файл с тем же путем изменился, старые строки этого файла удаляются, а файл парсится заново.
7. Новые строки добавляются к уже существующей итоговой таблице.
8. Результат сохраняется в Parquet, CSV и XLSX.

Флаг `--rebuild` отключает инкрементальность и полностью пересобирает итоговую таблицу из всех исходных файлов.

## `sales_by_counterparties`

Код: [sales_by_counterparties_pipeline.py](sales_by_counterparties_pipeline.py)

Источник:

`Данные/Продажи по контрагентам`

Результаты:

- `Данные/Итоговые таблицы/sales_by_counterparties.csv`
- `Данные/Итоговые таблицы/sales_by_counterparties.xlsx`
- `Данные/Итоговые таблицы/sales_by_counterparties.parquet`

Гранулярность: одна строка = одна строка номенклатуры внутри документа реализации или корректировки.

### Как парсится Excel

Исходный отчет имеет структуру:

1. контрагент;
2. договор;
3. документ продажи или корректировки;
4. номенклатура;
5. пары дневных колонок `Количество` и `Сумма`.

Пайплайн проходит по блокам:

```text
Контрагент
  Договор
    Реализация / Корректировка реализации
      Номенклатура
```

Для каждой строки номенклатуры и каждой непустой пары дневных колонок создается строка итоговой таблицы.

### Правило даты

`sale_date` сначала берется из дневной колонки Excel. Если дата документа `sales_doc_date` распарсилась и отличается от даты колонки, используется дата документа.

Это нужно для случаев, когда документ попадает в отчетную таблицу в техническую колонку без явной даты или в колонку, отличающуюся от даты документа.

### Основные поля

- `counterparty_raw`, `legal_entity`, `brand`, `store_location_raw`, `city_or_area` - контрагент и точка.
- `contract_raw`, `contract_number`, `contract_date` - договор.
- `sales_doc_raw`, `sales_doc_type`, `sales_doc_number`, `sales_doc_date` - документ продажи или корректировки.
- `sale_date`, `nomenclature`, `quantity`, `amount` - факт продажи.
- `period_label`, `period_start`, `period_end` - период из имени файла.
- `source_*` - технические поля происхождения строки.

## `sales_by_payment_counterparties`

Код: [sales_by_payment_counterparties_pipeline.py](sales_by_payment_counterparties_pipeline.py)

Источник:

`Данные/Продажи по контрагентам (по оплате)`

Результаты:

- `Данные/Итоговые таблицы/sales_by_payment_counterparties.csv`
- `Данные/Итоговые таблицы/sales_by_payment_counterparties.xlsx`
- `Данные/Итоговые таблицы/sales_by_payment_counterparties.parquet`

Гранулярность: одна строка = один платежный документ / одно поступление на расчетный счет.

### Как парсится Excel

Исходный отчет имеет структуру:

1. контрагент;
2. один или несколько договоров;
3. документы `Поступление на расчетный счет`;
4. дневные колонки с суммами оплат.

Пайплайн проходит по блокам:

```text
Контрагент
  Договор
    Поступление на расчетный счет
```

Для каждого платежного документа и каждой непустой дневной колонки создается строка итоговой таблицы.

### Правило даты

`payment_date` сначала берется из дневной колонки Excel. Если дата из текста документа `Поступление на расчетный счет № ... от ...` распарсилась и отличается от даты колонки, используется дата документа.

Служебные даты наружу не выводятся: в итоговой таблице остается только `payment_date`.

### Основные поля

- `counterparty_raw`, `legal_entity`, `brand`, `store_location_raw`, `city_or_area` - контрагент и точка.
- `contract_raw`, `contract_number`, `contract_date` - договор.
- `payment_doc_raw`, `payment_doc_number` - документ оплаты.
- `payment_date`, `amount` - факт оплаты.
- `period_label`, `period_start`, `period_end` - период из имени файла.
- `source_*` - технические поля происхождения строки.

## Как добавлять новые данные

Для продаж:

1. Положи новый `.xls` в `Данные/Продажи по контрагентам`.
2. Запусти:

```bash
python3 "Данные/Пайплайны обработки данных/sales_by_counterparties_pipeline.py"
```

Для оплат:

1. Положи новый `.xls` в `Данные/Продажи по контрагентам (по оплате)`.
2. Запусти:

```bash
python3 "Данные/Пайплайны обработки данных/sales_by_payment_counterparties_pipeline.py"
```

Для банковских выписок:

1. Положи новый `.xlsx` в `Данные/Выписки и данные`.
2. Запусти:

```bash
python3 "Данные/Пайплайны обработки данных/bank_statement_transactions_pipeline.py"
```

Если нужно пересобрать таблицу с нуля, добавь `--rebuild`.

## `bank_statement_transactions`

Код: [bank_statement_transactions_pipeline.py](bank_statement_transactions_pipeline.py)

Источник:

`Данные/Выписки и данные`

Результаты:

- `Данные/Итоговые таблицы/bank_statement_transactions.csv`
- `Данные/Итоговые таблицы/bank_statement_transactions.xlsx`
- `Данные/Итоговые таблицы/bank_statement_transactions.parquet`

Гранулярность: одна строка = одна банковская операция из табличной части выписки.

### Как парсится Excel

Исходная выписка имеет структуру:

1. шапка выписки с периодом, владельцем счета, остатками и оборотами;
2. двухстрочный заголовок таблицы;
3. строки операций.

Пайплайн находит первую строку, где первая колонка является датой формата `ДД.ММ.ГГГГ`, и дальше читает только операции. Колонки выписки переводятся в английские имена:

- `operation_date`, `document_number`, `debit`, `credit`;
- `counterparty_name`, `counterparty_inn`, `counterparty_kpp`;
- `counterparty_account`, `counterparty_bik`, `counterparty_bank_name`;
- `payment_purpose`, `debtor_code`, `document_type`.

Дополнительно считаются удобные аналитические поля:

- `amount = credit - debit`;
- `direction = credit` для поступлений и `direction = debit` для списаний.

### Проверка оборотов

Шапка выписки не попадает в итоговую таблицу. Из нее берутся только контрольные значения `Обороты дебет` и `Обороты кредит`.

После парсинга каждого файла пайплайн сверяет:

- сумму `debit` по строкам операций с `Обороты дебет`;
- сумму `credit` по строкам операций с `Обороты кредит`.

Если сверка не проходит, пайплайн останавливается с ошибкой.

### Основные поля

- `operation_date`, `document_number`, `document_type` - дата, номер и тип документа.
- `debit`, `credit`, `amount`, `direction` - суммы и направление движения денег.
- `counterparty_*` - реквизиты контрагента и его банка.
- `payment_purpose`, `debtor_code` - назначение платежа и код дебитора.
- `source_*` - технические поля происхождения строки.
